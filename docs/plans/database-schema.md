# Database Schema

## What this is

The whole product fits in a single PostgreSQL 16 database with about ten tables. Nothing exotic, no sharding, no read replicas in MVP — just well-indexed tables with clear ownership. This doc walks every table in plain English: what it stores, why it exists, which feature uses it, what's indexed, and which fields are JSONB and why.

The schema is small on purpose. Every feature in this docs set should be implementable by reading and writing these tables. If a new feature needs a new table, add it to this doc first and then build.

## Migration strategy

Migrations live in `backend/migrations/` as numbered SQL files: `001_users.sql`, `002_api_keys.sql`, and so on up to `010_*`. A tiny custom runner (a few dozen lines of Node) connects to the database, reads a `schema_migrations` table to find the highest applied number, and runs every newer file in order inside a transaction. There is no down migration story — forward-only with a fresh database for local resets.

In production, the runner runs as a one-shot Render job before each deploy of the backend. In dev, it runs on `npm run db:migrate`. Test runs use a separate database name and migrate from scratch each suite.

## The tables

### users

The root table. One row per signed-up account. Fields:

- `id` — UUID primary key, generated by the database.
- `email` — citext, unique. Stored case-insensitively so `Foo@bar.com` and `foo@bar.com` collide on signup.
- `password_hash` — bcrypt hash with cost 12. Never the plaintext.
- `plan` — text enum-ish: `free`, `starter`, `pro`. Default `free`. Drives credit grants and rate-limit ceilings.
- `credit_balance` — integer, default starts at the free-plan grant (e.g. 1000). Decremented per request, refilled on subscription renewal.
- `is_suspended` — boolean, default false. Set by an admin action; checked on every Bearer auth and every login. Suspended users get 403s.
- `created_at`, `updated_at` — timestamptz, default `now()`.

Indexes: unique on `email`. Read-heavy table (every Bearer auth resolves a user); writes are infrequent (signup, plan change, credit change).

Used by: `user-accounts-and-sessions.md`, `api-keys.md`, every authenticated endpoint, billing.

### api_keys

One row per API key a user has created. A user can have many keys; revoking one doesn't affect the others.

- `id` — UUID PK.
- `user_id` — FK to `users.id`, cascade on delete.
- `key_hash` — text, sha256 of the plaintext token, hex-encoded. Uniquely indexed so a Bearer lookup is one query.
- `key_prefix` — text, the first eight characters of the random suffix of the plaintext. Shown in the dashboard so users can identify keys without seeing them in full.
- `name` — text, user-provided label like "Production" or "Staging".
- `created_at` — timestamptz.
- `last_used_at` — timestamptz, nullable. Updated fire-and-forget on every successful Bearer auth (the request does not block on this write).
- `is_revoked` — boolean, default false.
- `revoked_at` — timestamptz, nullable.
- `expires_at` — timestamptz, nullable. Optional self-expiry; null means never expires.

Indexes: unique on `key_hash`, plus `(user_id, created_at desc)` for the dashboard list view.

Read-heavy on `key_hash` (every public API call hits it). Writes are rare except for `last_used_at`, which is high-volume but tolerant of being lossy.

Used by: `api-keys.md`, `transcript-endpoint.md` (via the Bearer middleware).

### subscriptions

Mirror of the user's Stripe subscription state. Not the source of truth — Stripe is — but a local cache keyed for fast reads.

- `user_id` — UUID, PK and FK. One subscription per user.
- `plan` — text, mirrors `users.plan` but updated by webhook.
- `stripe_customer_id` — text.
- `stripe_subscription_id` — text, nullable while in trial or canceled.
- `status` — text: `active`, `trialing`, `past_due`, `canceled`, `incomplete`. Mirrors Stripe's status field.
- `current_period_end` — timestamptz, nullable. Used to show the next renewal date and to trigger the credit grant on rollover.
- `created_at`, `updated_at` — timestamptz.

Indexes: PK on `user_id`. Index on `stripe_customer_id` for webhook dispatch.

Write-volume is whatever Stripe webhooks deliver — low. Read-volume is the dashboard billing page only.

Used by: billing feature (not in this docs set, but expected to exist), dashboard usage page.

### cached_transcripts

The canonical transcript cache. One row per `(video_id, language)` pair. This table is the single most important one for production — see the architecture overview for why.

- `video_id` — text, the 11-character YouTube id. Part of the composite PK.
- `language` — text, ISO 639-1 (e.g. `en`, `es`, `fr`, `auto`). Part of the composite PK. `auto` stores whatever language YouTube returned.
- `segments` — JSONB, an array of `{ start, dur, text }` objects. JSONB because the array structure is opaque to Postgres but queryable if we ever need to.
- `source` — text: `native` (from YouTube captions) or `whisper` (from OpenAI transcription).
- `metadata` — JSONB, holds `{ title, channel, thumbnail_url, duration }` from oEmbed. Stored alongside so a cache hit can return everything without a second fetch.
- `created_at` — timestamptz.

Indexes: composite PK on `(video_id, language)`. No other indexes — this table is only ever queried by exact key.

Read-heavy in production (every transcript request that misses Redis hits it). Write-once per video+language; never updated in place. If a video's captions change, we'd need to delete and refetch.

Used by: `transcript-endpoint.md`, `youtube-and-whisper.md`, dashboard transcript viewer.

### api_requests

Audit and analytics log. One row per `/v1/transcript` request that gets past authentication. Powers the usage charts in the dashboard and is the canonical source for billing reconciliation.

- `id` — UUID PK.
- `user_id` — FK to `users.id`.
- `api_key_id` — FK to `api_keys.id`. Lets us show "Production key vs Staging key" usage breakdowns.
- `video_id` — text, nullable (null for requests that errored before parsing).
- `language` — text, nullable.
- `format` — text: `json`, `text`, `srt`, `vtt`.
- `source` — text: `native`, `whisper`, `cache`. `cache` means served from the cache layer.
- `credit_cost` — integer. Zero for cache hits is allowed if pricing chooses; the column itself is just a record.
- `status_code` — smallint, the HTTP status returned.
- `latency_ms` — integer, wall-clock time from auth to response.
- `created_at` — timestamptz, default `now()`.

Indexes: `(user_id, created_at desc)` for the dashboard usage view. `(created_at)` for global analytics. No FK index on `api_key_id` is strictly required but useful.

Write-heavy (one row per public API call). Read patterns are bounded ranges (last 7 days, last 30 days). Old rows could be partitioned or rolled into aggregates later, but in MVP the table grows linearly and is fine for the first millions of rows.

Used by: `transcript-endpoint.md`, dashboard usage page.

### credit_transactions

Append-only ledger of every credit movement. One row per debit or credit, with the post-transaction balance for auditability.

- `id` — UUID PK.
- `user_id` — FK to `users.id`.
- `delta` — integer. Negative for debits (transcript request), positive for credits (subscription renewal, manual grant).
- `reason` — text: `request`, `grant_signup`, `grant_renewal`, `grant_manual`, `refund`. Free-form enough to add new reasons without a migration.
- `balance_after` — integer. The user's `credit_balance` immediately after this row was written. Lets us reconstruct balance over time without summing.
- `created_at` — timestamptz.

Indexes: `(user_id, created_at desc)` for the activity tab in the dashboard.

Write-heavy and append-only. Never updated, never deleted. Pairs with the `users.credit_balance` update inside the same transaction so balance can never drift from the ledger.

Used by: `transcript-endpoint.md` (debit), billing (credit), dashboard usage page.

### webhook_events

Stripe webhook idempotency table. Every Stripe event has a unique `evt_…` id; we record it here the first time we process the event and reject duplicates.

- `id` — text PK, the Stripe event id.
- `type` — text, the event type for debugging.
- `received_at` — timestamptz.

That's it. No FK to anything, no indexes beyond the PK. Written once per webhook delivery, never read except by an admin debugging.

Used by: billing feature.

### sessions (optional, may not exist)

The dashboard uses a stateless JWT in a cookie, so a sessions table is not strictly needed. If we ever want server-side revocation of a logged-in session, we'd add a `sessions` table with `(id, user_id, created_at, revoked_at)` and check it in the cookie middleware. For MVP, we omit it — logout just clears the cookie client-side, and the seven-day expiry caps the blast radius.

### credit_grants (optional, may not exist)

For very simple billing (one renewal per period, fixed grant), we don't need a separate `credit_grants` table — the renewal logic just inserts a `credit_transactions` row with reason `grant_renewal`. If we add stacking promo grants or variable-amount grants in the future, a `credit_grants` table tracking `(user_id, amount, expires_at, source)` would make sense. Not in MVP.

## JSONB, why

Two columns are JSONB: `cached_transcripts.segments` and `cached_transcripts.metadata`.

`segments` is an array of `{ start, dur, text }` objects. The shape is fixed and we never query individual elements from SQL — the whole blob is read at once and serialized to the response. JSONB is used over a separate `transcript_segments` table because (a) segments are always read together with their parent transcript, (b) we never want to update one segment, (c) a single row read is faster than reassembling an array of joined rows, and (d) a typical transcript can have hundreds of segments.

`metadata` is a small object with title, channel, thumbnail URL, and duration. It's stored as JSONB rather than as separate columns so we can add fields (e.g. publish date, view count) without a migration. The structure is small enough that the lack of column-level indexing doesn't matter.

## Read-heavy vs write-heavy

Quick summary so capacity planning is easy:

- **Read-heavy**: `users` (every auth), `api_keys` (every Bearer call), `cached_transcripts` (every transcript request that misses Redis).
- **Write-heavy**: `api_requests` (every public API call), `credit_transactions` (every debit and credit), `cached_transcripts` (every fresh fetch — but each video+language is only written once).
- **Mostly idle**: `subscriptions`, `webhook_events`. Webhook traffic is low.

Postgres handles all of this on a single small Neon instance comfortably for the first several thousand active users. The first thing to fall over at scale will be `api_requests` row count — at which point partitioning by month or rolling old rows into aggregates is the lever to pull.

- Insert a test user, an api key for that user, and a cached transcript; confirm joins resolve correctly.
- Drop the database, re-run migrate, confirm the schema is identical (idempotent on a clean DB).
- For each table listed above, confirm the columns and indexes match this doc by inspecting `\d+ <table>` in psql.
